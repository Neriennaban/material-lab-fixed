## 1. Обзор и цели

### 1.1. Проблема

В текущей версии виртуальной металлографической лаборатории студенты могут получить доступ к правильным ответам через JSON metadata файлы:

- Точный состав материала (который они должны рассчитать)
- Точные доли фаз (которые они должны измерить)
- Марка стали (которую они должны определить)
- Система материала (которую они должны идентифицировать)

Студент может просто открыть `lab_sample.json` в текстовом редакторе и списать все ответы, что делает лабораторную работу бессмысленной.

### 1.2. Цели решения

1. **Защита ответов:** Правильные ответы должны быть зашифрованы и доступны только преподавателю
2. **Разделение режимов:** Четкое разделение интерфейса для студентов и преподавателей
3. **Верификация подлинности:** Невозможность подделки образцов или результатов
4. **Удобство использования:** Простой и понятный интерфейс для обычных пользователей
5. **Максимальная безопасность:** Криптографическая защита уровня RSA-2048

### 1.3. Что студент ДОЛЖЕН видеть

Согласно методичкам ЛР1 и ЛР2, студент должен иметь доступ к:

✅ **Доступно студенту:**
- Изображение шлифа
- Состав материала (Fe, C) - дано в условии задачи
- Параметры термообработки (температуры, время)
- Параметры подготовки шлифа (шлифовка, полировка, травление)
- Инструменты для измерений (линейка, секущие)

❌ **Скрыто от студента:**
- Точные доли фаз (должен измерить сам)
- Марка стали (должен определить по расчетам)
- Система материала (может быть подсказкой)
- Правильные ответы для проверки

## 2. Архитектура системы безопасности

### 2.1. Компоненты

**RSA ключи (2048-bit):**
- **Публичный ключ:** встроен в приложение (hardcoded или в config файле)
- **Приватный ключ:** хранится у преподавателя в файле `teacher_private_key.pem`
- **Генерация:** один раз при первой настройке системы через утилиту

**Файловая структура экспорта:**
```
lab_package_YYYYMMDD_HHMMSS/
├── lab_sample.png                    # изображение шлифа
├── lab_sample_student.json           # данные для студента + цифровая подпись
├── lab_sample_answers.enc            # зашифрованные ответы (RSA + AES)
├── lab_sample_manifest.json          # манифест пакета
├── thermal_program.png               # график термопрограммы
└── thermal_program.csv               # точки термопрограммы
```

**lab_sample_student.json** (для студента):


### 2.3. Криптографическая схема

**Гибридное шифрование (RSA + AES):**

1. Генерируем случайный AES-256 ключ
2. Шифруем JSON данные AES-256-GCM (быстро, для больших данных)
3. Шифруем AES ключ публичным RSA ключом (безопасно)
4. Сохраняем: `[RSA encrypted AES key][AES encrypted data][GCM tag]`

**Цифровая подпись student.json:**

1. Создаем канонический JSON (sorted keys, no whitespace)
2. Подписываем SHA256 хеш приватным RSA ключом
3. Добавляем подпись в поле `digital_signature`
4. Проверка: публичный ключ (встроен в приложение) проверяет подпись

**Верификация целостности:**

1. SHA256 хеш изображения записывается в student.json и answers.enc
2. При проверке: пересчитываем хеш изображения и сравниваем
3. Если хеши не совпадают → образец подделан

## 5. Процесс верификации

### 5.1. При генерации образца

**Последовательность действий в `_export_lab_package()`:**

1. Генерируем изображение → `lab_sample.png`
2. Вычисляем `SHA256(image)` → `image_hash`
3. Собираем данные для студента:
   - Состав, термопрограмма, параметры обработки
   - `image_sha256 = image_hash`
   - `sample_id`, `timestamp`, `seed`
4. Создаем канонический JSON и подписываем приватным ключом
5. Сохраняем `lab_sample_student.json` с подписью
6. Собираем ответы для преподавателя:
   - Доли фаз из `phase_model_report`
   - Марка стали (вычисляем из C)
   - `image_sha256 = image_hash` (тот же)
7. Шифруем ответы гибридным методом (RSA + AES)
8. Сохраняем `lab_sample_answers.enc`

### 5.2. При проверке образца

**Последовательность действий в функции верификации:**

1. Загружаем `lab_sample.png`
2. Вычисляем `SHA256(image)` → `current_hash`
3. Загружаем `lab_sample_student.json`
4. Извлекаем `image_sha256` и `digital_signature`
5. Проверяем подпись публичным ключом:
   - Если невалидна → ❌ "Данные были изменены"
6. Сравниваем `current_hash == image_sha256`:
   - Если не совпадают → ❌ "Изображение было изменено"
7. Загружаем приватный ключ преподавателя
8. Расшифровываем `lab_sample_answers.enc`
9. Извлекаем `image_sha256` из ответов
10. Сравниваем все три хеша:
    - `current_hash == student.image_sha256 == answers.image_sha256`
11. Если все проверки пройдены → ✅ "Образец подлинный"

### 7.3. API криптографического модуля

**crypto_manager.py:**

```python
def generate_rsa_keypair() -> tuple[bytes, bytes]:
    """Генерирует пару RSA ключей (приватный, публичный)"""

def compute_image_hash(image_path: Path) -> str:
    """Вычисляет SHA256 хеш изображения"""

def sign_data(data: dict, private_key: bytes) -> str:
    """Создает цифровую подпись для данных"""

def verify_signature(data: dict, signature: str, public_key: bytes) -> bool:
    """Проверяет цифровую подпись"""

def encrypt_answers(answers: dict, public_key: bytes) -> bytes:
    """Шифрует ответы гибридным методом (RSA + AES)"""

def decrypt_answers(encrypted: bytes, private_key: bytes) -> dict:
    """Расшифровывает ответы"""
```

**verification.py:**

```python
def verify_sample_integrity(
    image_path: Path,
    student_json_path: Path,
    answers_enc_path: Path,
    private_key_path: Path
) -> VerificationResult:
    """Полная проверка подлинности образца"""

@dataclass
class VerificationResult:
    is_valid: bool
    image_authentic: bool
    data_authentic: bool
    signature_valid: bool
    hashes_match: bool
    error_message: str | None
    answers: dict | None
```

## 10. Риски и ограничения

### 10.1. Риски

1. **Потеря приватного ключа преподавателем**
   - Решение: создать процедуру восстановления/перегенерации ключей

2. **Студенты могут обмениваться образцами**
   - Решение: генерировать уникальные образцы для каждого студента

3. **Сложность для преподавателей**
   - Решение: максимально упростить UI, создать подробные инструкции

### 10.2. Ограничения

1. **Размер зашифрованных файлов**
   - Answers.enc будет ~1-2 KB (незначительно)

2. **Производительность**
   - RSA операции медленные, но выполняются редко (только при экспорте/проверке)

3. **Обратная совместимость**
   - Старые образцы без шифрования не будут работать с новой системой
   - Решение: добавить режим совместимости для чтения старых файлов
